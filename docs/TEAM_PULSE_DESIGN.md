# 投资汇报助手 MVP - 产品方案

**版本**: v2.2 (MVP)  
**日期**: 2026-04-20  
**状态**: 待确认

---

## 一、核心价值主张

**一句话**：通过 AI 智能追问，挖掘项目真实状况和潜在风险，生成结构化报告，支持投资决策。

**核心价值**：
- 员工可能有意无意忽略关键信息 → AI 追问挖掘出来
- 风险点描述模糊 → AI 追问量化影响
- 问题只摆在那里 → AI 追问推动形成解决方案

**关键洞察**：如果问题固定，员工会摸清套路，做出针对性答复。**智能追问的核心是动态适应**——根据员工的回答方式、内容深度、回避程度，实时调整追问策略。

---

## 二、MVP 范围

**只做一个场景**：投后管理周报

**理由**：
1. 最高频（每周）
2. 价值最直接（识别风险、支持决策）
3. 流程最简单（定时触发 → 追问 → 报告）
4. 不依赖外部系统（出差审批等）

**MVP 流程**：

每周五 17:00 → 发送提醒 → 员工回复项目名称 → AI 智能追问（3-5轮）→ 生成报告 → 发送给上级

---

## 三、智能追问引擎（核心能力）

### 3.1 设计理念

**核心原则：问题不固定 + 顺序不固定 + 轮次不固定**

| 维度 | 设计原则 | 防套路机制 |
|------|----------|------------|
| 追问目标 | 5个目标池（非固定顺序） | 根据回答动态选择下一个目标 |
| 具体问题 | 动态生成 | 每次措辞、角度不同 |
| 追问顺序 | 非线性 | 随机因素 + 内容驱动 |
| 追问轮次 | 3-7轮动态 | 质量高提前结束，风险大深入追问 |
| 语气风格 | 多样化 | 随机选择不同风格 |

**关键洞察**：员工摸清套路需要"可预测性"。只要打破可预测性（顺序、轮次、措辞），就无法针对性准备。

---

### 3.2 追问目标池（非固定顺序）

**5个追问目标，但顺序动态调整**：

| 目标 | 核心关注点 | 触发条件 |
|------|------------|----------|
| 项目进展 | 里程碑、进度、关键数据 | 默认首个或回答中提到进展 |
| 问题识别 | 困难、风险、异常情况 | 回答中提到问题或默认第二轮 |
| 风险深挖 | 具体数据、根本原因、影响范围 | 发现问题后自动深入 |
| 解决方案 | 应对措施、责任人、时间节点 | 问题明确后追问方案 |
| 决策建议 | 建议、支持需求 | 收尾阶段 |

**动态顺序逻辑**：

```
IF 回答中主动提到问题 → 跳转到"风险深挖"
IF 回答中提到风险已解决 → 跳转到"解决方案"
IF 回答质量高 + 无新问题 → 可跳过某些目标
IF 随机因素触发 → 插入"确认性问题"（防止套路）
```

**随机因素示例**：
- 20% 概率插入一个"回马枪"问题（回到之前的话题）
- 15% 概率改变追问角度（从数据角度转为时间角度）
- 10% 概率改变语气风格（从正式转为轻松）

---

### 3.3 动态追问生成机制

#### 3.3.1 核心架构

智能追问引擎 = 回答分析器 + 追问决策器 + 问题生成器 + 反套路检测器

#### 3.3.2 回答分析器（简化版）

**功能**：分析员工回答的质量，判断是否需要追问

**MVP 分析维度**（只保留最关键的2个）：

| 维度 | 判断标准 | 分析方法 |
|------|----------|----------|
| 具体性 | 是否有具体数据/细节 | 数字检测 + 量词检测 |
| 新信息 | 是否有意外的关键信息 | 关键词提取（风险、问题、异常） |

**简化原因**：MVP 阶段不需要复杂的语义分析，关键词 + 数字检测足够识别回答质量。

#### 3.3.3 追问决策器

**功能**：根据回答分析结果，决定追问策略

**决策逻辑**：

```
IF 回答具体性 < 0.5 → 追问策略 = "追问具体数据"
ELIF 发现新信息（风险/问题关键词） → 追问策略 = "深入挖掘新信息"
ELIF 已达到当前目标 AND 轮次 >= 3 → 追问策略 = "进入下一目标"
ELIF 轮次 >= 7 → 追问策略 = "结束追问"
ELSE → 追问策略 = "继续当前目标"
```

**轮次动态调整**：
- 最少 3 轮（回答质量高可提前结束）
- 最多 7 轮（防止疲劳）
- 平均 5 轮

#### 3.3.4 问题生成器

**功能**：根据追问策略，生成具体问题

**输入**：当前追问目标、追问策略、对话历史、历史汇报、追问模板库、随机风格参数

**输出**：具体问题文本

**随机性注入**：
- 风格参数：正式/轻松/直接/委婉（随机选择）
- 角度参数：数据角度/时间角度/人员角度（随机选择）
- 措辞参数：多种等效表达随机选择

#### 3.3.5 反套路检测器（MVP 简化版）

**功能**：检测员工是否在用套路回答

**检测维度**：

| 维度 | 检测方法 | 触发动作 |
|------|----------|----------|
| 回答长度 | 连续 3 次回答 < 20 字 | 插入追问："能详细说说吗？" |
| 关键词堆砌 | 回答包含多个关键词但无具体数据 | 追问具体数据 |
| 历史重复 | 与上周回答相似度 > 80% | 追问："这周有什么新变化吗？" |
| 回避模式 | 连续 2 次不回答核心问题 | 换个角度追问 |

**MVP 简化**：只检测上述 4 种明显套路，不做复杂的语义分析。

---

### 3.4 追问模板库（参考用，非固定）

每个追问目标下，预设多种追问方式模板，LLM 根据实际情况选择或改编：

**第1轮：项目进展**

| 追问场景 | 模板示例 |
|----------|----------|
| 了解里程碑 | "本周完成了哪些里程碑？和计划对比如何？" |
| 追问进度 | "进度是超前还是滞后？大概差多少？" |
| 追问数据 | "有关键数据可以分享吗？" |
| 对比历史 | "和上周/上月相比，有什么变化？" |

**第2轮：问题识别**

| 追问场景 | 模板示例 |
|----------|----------|
| 直接询问 | "目前遇到什么困难或问题吗？" |
| 追问异常 | "你提到的这个情况，能具体说说吗？" |
| 追问风险 | "有没有什么潜在风险需要关注？" |

**第3轮：风险深挖**

| 追问场景 | 模板示例 |
|----------|----------|
| 量化影响 | "这个问题影响多大？有具体数据吗？" |
| 追问原因 | "根本原因是什么？" |
| 追问范围 | "影响范围有多大？" |

**第4轮：解决方案**

| 追问场景 | 模板示例 |
|----------|----------|
| 追问方案 | "有什么应对措施吗？" |
| 追问责任 | "谁来负责？什么时候能解决？" |

**第5轮：决策建议**

| 追问场景 | 模板示例 |
|----------|----------|
| 投资建议 | "你的建议是追加、持有还是退出？" |
| 支持需求 | "需要公司提供什么支持？" |

**模板使用原则**：
- 模板仅供参考，LLM 可根据实际情况改编
- 问题要自然，像真人对话
- 不要让员工觉得在审问

### 3.5 LLM Prompt 设计

#### 3.5.1 追问生成 Prompt

你是投资汇报助手，正在进行第{round_num}轮追问。

【当前追问目标】
{current_target}

【对话历史】
{conversation_history}

【员工最新回答】
{employee_answer}

【历史汇报参考】
{last_week_report}

【随机风格参数】
- 语气风格：{style}（正式/轻松/直接/委婉）
- 追问角度：{angle}（数据角度/时间角度/人员角度）

【分析任务】
1. 判断员工回答是否有具体数据
2. 识别是否有风险/问题关键词
3. 检测是否与历史汇报重复

【追问策略】
根据分析结果，选择追问策略：
- 追问细节：回答太笼统
- 深入挖掘：发现新问题/风险
- 历史对比：与上周重复
- 进入下一轮：当前目标已达成

【追问模板参考】
{relevant_templates}

【输出要求】
生成一个自然、具体的追问，像真人对话。使用{style}语气，从{angle}角度提问。

---

#### 3.5.2 报告生成 Prompt

你是投资汇报助手，需要生成结构化周报。

【对话记录】
{full_conversation}

【报告要求】
1. 项目进展：里程碑、进度、关键数据
2. 问题与风险：具体问题、影响评估
3. 解决方案：应对措施、责任人
4. 投资建议：建议方向、支持需求

【输出格式】
Markdown 格式，结构清晰，数据具体。

### 3.6 质量调优机制

#### 3.6.1 质量评估指标

| 指标 | 定义 | 评估方式 |
|------|------|----------|
| 信息完整度 | 报告是否覆盖关键维度 | 上级评分（1-5） |
| 风险识别率 | 是否发现真实风险 | 后续事件验证 |
| 追问效率 | 多少轮达到目标 | 平均轮数统计 |
| 员工接受度 | 员工是否觉得自然 | 员工反馈 |

#### 3.6.2 调优方式

**短期调优（每周）**：
- 分析本周所有对话
- 找出追问效果差的案例
- 调整 Prompt 或模板

**长期调优（每月）**：
- 统计质量指标趋势
- A/B 测试不同策略
- 更新模板库

#### 3.6.3 反馈收集

- 上级可对报告评分（1-5星）
- 员工可反馈追问是否合理
- 系统记录追问轮数、时间

## 四、数据模型

**MVP 只用一张表**：

| 字段 | 类型 | 说明 |
|------|------|------|
| record_id | 文本 | 记录ID |
| project_name | 文本 | 项目名称 |
| employee_id | 文本 | 员工ID |
| employee_name | 文本 | 员工姓名 |
| week | 文本 | 周次（如 2026-W16） |
| conversation | 多行文本 | 对话记录（JSON） |
| report | 多行文本 | 生成的报告 |
| status | 单选 | 状态：待开始/进行中/已完成 |
| created_time | 创建时间 | 记录创建时间 |
| completed_time | 创建时间 | 报告完成时间 |

**历史对比实现**：
- 查询条件：同一 employee_id + project_name + 上周 week
- 对比维度：回答相似度、关键词变化、风险点变化
- MVP 简化：只对比上周，不做多周趋势分析

## 五、实现计划

### 5.1 技术栈

| 组件 | 选择 | 理由 |
|------|------|------|
| 平台 | 飞书多维表格 | 零成本、易配置、团队熟悉 |
| 触发 | 飞书机器人 | 定时提醒 + 消息交互 |
| LLM | DeepSeek / Claude | 成本低、效果好 |
| 存储 | 飞书多维表格 | 无需额外数据库 |

### 5.2 开发步骤

**Day 1：基础搭建**
- 创建飞书多维表格
- 配置飞书机器人
- 实现定时提醒

**Day 2：追问引擎**
- 实现追问目标框架
- 实现 Prompt 模板
- 对接 LLM API

**Day 3：报告生成**
- 实现报告生成逻辑
- 实现报告发送
- 测试完整流程

### 5.3 MVP 验证指标

| 指标 | 目标 | 验证方式 |
|------|------|----------|
| 完成率 | >80% 员工完成汇报 | 统计完成人数 |
| 信息质量 | 上级评分 >3.5 | 收集上级反馈 |
| 风险发现 | 至少发现1个真实风险 | 后续事件验证 |

## 七、风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 员工不配合 | 中 | 高 | 强调价值、简化流程 |
| LLM 回答质量差 | 低 | 中 | Prompt 优化、模板库 |
| 飞书 API 限制 | 低 | 中 | 限流控制、备用方案 |
