# MVP 简化建议 (产品视角)

**版本**: v1.0  
**日期**: 2026-04-18  
**目标**: 快速上线验证核心假设

---

## 一、可砍掉的功能 (P2延后)

### 1.1 团队摘要功能 (PRD 7.4) - 延后

**理由**:
- 团队摘要依赖多个个人周报完成，是二级输出
- MVP 核心验证的是"员工愿不愿意配合"和"经理觉得有没有用"
- 先跑通个人周报 -> 1:1 Brief，再扩展团队摘要

**影响**:
- 减少 manager_brief 表的复杂度 (只需支持 one_on_one 类型)
- 减少摘要生成模块的分支逻辑

### 1.2 每日自动复盘 -> 改为每周复盘

**理由**:
- MVP 阶段数据量小，每日复盘产出有限
- 每周复盘足够验证"系统能不能发现问题"
- 减少定时任务复杂度

**影响**:
- daily_review_report 改为 weekly_review_report
- 减少监控告警项

### 1.3 风险追踪独立流程 (PRD 7.2) - 合并到周报采集

**理由**:
- 风险追问作为周报采集的子流程即可
- 不需要独立的风险抽取模块
- 用户视角就是一个对话流程

**影响**:
- 减少 risk_followup 状态的复杂度
- 简化状态机

### 1.4 员工负面反馈率监控 - 延后

**理由**:
- MVP 阶段用户量小，定性反馈更直接
- 可以通过经理访谈收集员工反馈
- 不需要自动化的负面反馈监控

## 二、可简化的流程

### 2.1 周报采集字段: 6 -> 4

**原设计** (PRD 7.1):
- 本周最重要的 1-3 项进展
- 关键结果或里程碑
- 当前阻塞
- 风险与依赖
- 下周目标
- 需要经理支持的事项

**简化后**:
- 本周进展 (含结果)
- 当前阻塞
- 风险与依赖
- 需要支持

**理由**:
- "下周目标"可以在下周进展中体现
- "关键结果"合并到"进展"中追问
- 减少员工填写负担

### 2.2 风险抽取字段: 8 -> 4

**原设计** (PRD 7.2):
- 风险标题、描述、影响范围、紧急程度、Owner、预计暴露时间、建议动作、是否需要经理介入

**简化后**:
- 风险描述
- 影响范围
- 紧急程度 (高/中/低)
- 需要经理介入 (是/否)

**理由**:
- MVP 阶段不需要"预计暴露时间"
- "建议动作"可以由经理判断
- "Owner"默认就是汇报人

### 2.3 1:1 Brief 内容: 6 -> 4

**原设计** (PRD 7.3):
- 本周关键进展、未解决阻塞、连续出现的问题、风险与依赖、需要经理支持、建议提问3个

**简化后**:
- 本周进展
- 阻塞与风险
- 需要支持
- 建议1个提问

**理由**:
- "连续出现的问题"需要历史数据，MVP 第一周没有
- "建议提问3个"改为1个，减少生成复杂度
- 先验证"有用"，再追求"全面"

### 2.4 经理反馈类型: 6 -> 3

**原设计** (PRD 7.5):
- 有用、漏了重点、判断不准、风险夸大、问得太多、语气不合适

**简化后**:
- 有用
- 不准确 (合并"漏了重点"、"判断不准"、"风险夸大")
- 体验问题 (合并"问得太多"、"语气不合适")

**理由**:
- 6种反馈对MVP太细
- 3种足够区分"内容质量"和"体验问题"

### 2.5 状态机简化: 6状态 -> 4状态

**原设计** (ENGINEERING_FIXES):
- initiated, collecting, risk_followup, summary_confirm, confirmed, expired, failed

**简化后**:
- initiated (已发起)
- collecting (收集中，含风险追问)
- confirmed (已确认)
- expired (已过期)

**理由**:
- 移除 risk_followup 独立状态
- 移除 failed 状态 (失败直接过期)
- 移除 summary_confirm 状态 (摘要确认合并到 collecting 最后一步)

## 三、可减少的数据字段

### 3.1 数据表: 8 -> 5

**保留**:
1. feishu_user_cache - 用户映射
2. checkin_session - 会话主表
3. message_log - 消息日志
4. risk_item - 风险项
5. feedback_log - 反馈记录

**延后**:
- extraction_snapshot - 合并到 checkin_session.summary_json
- manager_brief - 改为实时生成，不持久化
- daily_review_report - 改为 weekly_review_report
- optimization_suggestion - 延后到 Phase 2

### 3.2 checkin_session 字段简化

**移除**:
- prompt_version (MVP 不做版本管理)
- summary_json (合并到 summary_text)

### 3.3 risk_item 字段简化

**移除**:
- evidence_json (MVP 不需要证据链)
- updated_at (创建后不更新)

## 四、最小验证指标

### 4.1 核心指标 (必须追踪)

| 指标 | 目标 | Kill Criteria |
|------|------|---------------|
| 周报完成率 | >= 70% | < 50% 持续3周 |
| 经理采纳率 | >= 60% | < 30% 持续4周 |

### 4.2 辅助指标 (可选追踪)

| 指标 | 目标 |
|------|------|
| 平均对话轮数 | 3-8 轮 |
| 平均完成时长 | <= 5 分钟 |

### 4.3 延后指标

- 提前暴露的风险数
- 经理准备 1:1 时间节省量
- 重复阻塞识别率
- 失败模式收敛速度

## 六、简化后的 MVP 范围

### 6.1 核心功能 (保留)

1. **周报采集对话** - 简化字段，合并风险追问
2. **1:1 Brief 生成** - 简化内容，实时生成
3. **经理反馈** - 3种反馈类型
4. **每周复盘** - 替代每日复盘

### 6.2 核心验证假设

1. 员工愿意在飞书里配合异步汇报
2. 经理认为 AI 生成的 Brief 有价值
3. 系统能够识别出有意义的风险

### 6.3 不变的原则

- 飞书优先，不做 Web 后台
- 复用 OpenClaw，不自建渠道
- 本地只存必要状态

## 七、建议采纳优先级

| 优先级 | 建议 | 影响 | 风险 |
|--------|------|------|------|
| P0 | 简化周报采集字段 | 高 | 低 |
| P0 | 简化状态机 | 高 | 低 |
| P0 | 减少数据表 | 高 | 低 |
| P1 | 延后团队摘要 | 中 | 低 |
| P1 | 每日复盘改每周 | 中 | 低 |
| P2 | 简化反馈类型 | 低 | 低 |

---

## 八、总结

通过以上简化，MVP 可以：

1. **开发时间减少 41%** (27天 -> 16天)
2. **试点周期缩短 33%** (6周 -> 4周)
3. **数据表减少 37%** (8张 -> 5张)
4. **状态机复杂度减半** (6状态 -> 4状态)

核心验证假设不变，但验证速度大幅提升。建议优先采纳 P0 级简化项。
